# **Dispatch OS — Operating Loop Rules**

## **1\. Purpose**

Dispatch OS exists to turn messy business inputs into structured operating memory, executable work, measurable value, and repeatable improvement.

The system should not be built around a specific industry.

It should be built around the universal mechanics of how businesses operate:

1. Something enters the business.  
2. The business interprets it in context.  
3. A decision is made.  
4. Work is created or updated.  
5. Evidence is captured.  
6. Value or impact is measured.  
7. The system learns whether to repeat, deviate, or innovate.

This loop is the foundation of Dispatch OS.

---

# **2\. The Core Loop**

The Dispatch OS loop is:

**Input → Context → Interpretation → Decision → Action → Evidence → Value → Feedback → Adaptation**

Each stage must be represented in the product, database, agent logic, and UI.

No stage should be skipped.

---

# **3\. Input**

## **Definition**

An input is anything the business receives, observes, uploads, exports, forwards, creates, or experiences.

Inputs are the raw material of the operating system.

## **Input Examples**

Inputs may include:

* emails  
* documents  
* PDFs  
* spreadsheets  
* CSV exports  
* screenshots  
* notes  
* forms  
* messages  
* transactions  
* appointments  
* reports  
* customer requests  
* employee updates  
* system alerts  
* API events  
* calendar events  
* financial records  
* operational exceptions  
* human observations

## **Input Rule**

The system should accept imperfect inputs.

Dispatch OS should not require perfect integrations to create value.

If a user can upload it, forward it, export it, paste it, connect it, or describe it, the system should be able to intake it.

## **Input States**

Every input should move through states:

1. Received  
2. Parsed  
3. Classified  
4. Matched  
5. Interpreted  
6. Proposed  
7. Approved  
8. Converted  
9. Archived or rejected

## **Input Storage Principle**

Raw input must be preserved.

The system should distinguish between:

* raw input  
* extracted data  
* interpreted meaning  
* human-approved truth

This distinction is critical for trust, auditability, and agent safety.

---

# **4\. Automation Key**

## **Definition**

An automation key is the connection point between an input and the operating loop.

It answers:

Why does this input matter, and what should it trigger?

An automation key can be:

* a source  
* a type  
* a keyword  
* a rule  
* a field value  
* a status  
* a deadline  
* a missing item  
* a variance  
* a risk signal  
* an opportunity signal  
* a recurring pattern  
* a human instruction  
* an agent classification

## **Automation Key Examples**

* A new document is uploaded.  
* A required field is missing.  
* A deadline is approaching.  
* A customer request is unresolved.  
* A report shows a variance.  
* A status has not changed in too long.  
* A recurring issue appears again.  
* An agent identifies a gap.  
* A human flags an item for review.

## **Automation Key Rule**

Automation keys should not be hardcoded to one industry.

They should be configurable.

A configuration should define:

* what input types matter  
* what conditions matter  
* what thresholds matter  
* what exceptions matter  
* what opportunities matter  
* what should trigger work  
* what should trigger review  
* what should trigger reporting

---

# **5\. Context**

## **Definition**

Context is the business-specific operating memory that allows the system to understand what an input means.

The same input can mean different things depending on the business.

Context includes:

* business model  
* goals  
* roles  
* customers  
* assets  
* services  
* obligations  
* workflows  
* rules  
* constraints  
* standards  
* preferences  
* definitions  
* historical patterns  
* risk tolerances  
* value drivers  
* escalation logic  
* approval requirements

## **Context Rule**

The Core OS should not assume context.

Context must be configured, learned, or approved.

## **Context Sources**

Context can come from:

* onboarding questions  
* uploaded documents  
* existing SOPs  
* user-created rules  
* previous work items  
* historical events  
* human corrections  
* agent summaries  
* imported records  
* management preferences  
* operating standards

## **Context Versioning**

Context should be versioned.

When business logic changes, the system should preserve:

* what changed  
* who changed it  
* when it changed  
* why it changed  
* what workflows it affects

---

# **6\. Configuration**

## **Definition**

Configuration is how Dispatch OS becomes specific without becoming hardcoded.

The Core OS provides universal primitives.

Configuration defines how those primitives behave for a business.

## **Configuration Owns**

Configuration may define:

* vocabulary  
* entity types  
* input types  
* workflow templates  
* checklist templates  
* rule sets  
* automation keys  
* evidence requirements  
* approval requirements  
* escalation logic  
* dashboards  
* reports  
* agent instructions  
* metrics  
* outcome definitions  
* value assumptions

## **Configuration Rule**

The product should be generic at the primitive level and specific at the configuration level.

The Core OS should not contain industry nouns as hardcoded architecture.

Industry/business-specific meaning belongs in configuration.

---

# **7\. Interpretation**

## **Definition**

Interpretation is the process of turning an input into possible meaning.

Interpretation asks:

* What is this?  
* What does it relate to?  
* Is it new?  
* Is it complete?  
* Is it accurate?  
* Is it normal?  
* Is it an exception?  
* Is it an opportunity?  
* Does it create risk?  
* Does it trigger work?  
* Does it require approval?  
* Does it update context?  
* Does it affect value?

## **Interpretation Sources**

Interpretation can come from:

* deterministic rules  
* agents  
* user review  
* system matching  
* historical patterns  
* configuration logic

## **Interpretation Rule**

Interpretation is not truth.

Agent interpretation must be staged as a proposal unless the configuration explicitly allows automation.

The system must distinguish:

* machine-suggested  
* rule-generated  
* human-approved  
* system-confirmed  
* rejected

---

# **8\. Decision**

## **Definition**

A decision is the selected response to an interpreted input.

A decision answers:

What should happen next?

## **Decision Types**

A decision may create:

* no action  
* a work item  
* a note  
* an evidence item  
* a document request  
* a data cleanup task  
* a review task  
* an escalation  
* an approval request  
* a report update  
* a context update  
* a workflow change  
* an agent run  
* a market/outcome output

## **Decision Rule**

Important decisions should be human-in-the-loop by default.

Automation should expand only after repeated evidence shows the loop is reliable.

---

# **9\. Action**

## **Definition**

An action is the execution object created by a decision.

The universal action object is the work item.

A work item represents something the business needs to do, review, prove, fix, decide, or monitor.

## **Work Item Fields**

A work item should support:

* title  
* description  
* owner  
* status  
* priority  
* due date  
* source input  
* related entities  
* related documents  
* checklist  
* notes  
* evidence  
* decision source  
* approval status  
* agent involvement  
* audit events  
* expected outcome  
* measured result

## **Action Rule**

Every meaningful interpretation should either:

1. create work,  
2. update existing work,  
3. update context,  
4. update evidence,  
5. update a report,  
6. or be intentionally dismissed.

Nothing should disappear without a recorded disposition.

---

# **10\. Evidence**

## **Definition**

Evidence is proof that something happened, changed, was reviewed, was approved, was completed, or was intentionally rejected.

Evidence may include:

* uploaded files  
* documents  
* notes  
* timestamps  
* checklist completion  
* photos  
* approvals  
* comments  
* source records  
* before/after state  
* user actions  
* agent outputs  
* human review  
* exception explanation

## **Evidence Rule**

Evidence is a core primitive, not an attachment afterthought.

Every important work item should have evidence requirements defined by configuration.

Evidence should answer:

* What happened?  
* Who did it?  
* When did it happen?  
* Why did it happen?  
* What source supports it?  
* Who approved it?  
* What changed as a result?

---

# **11\. Value**

## **Definition**

Value is the measurable reason the work matters.

Dispatch OS should not only track activity.

It should connect activity to business impact.

## **Value Types**

Value can include:

* time saved  
* revenue created  
* revenue protected  
* cost reduced  
* risk reduced  
* error avoided  
* faster completion  
* better compliance  
* improved customer experience  
* improved transferability  
* improved reporting  
* better staff capacity  
* reduced founder dependency  
* cleaner data  
* increased readiness  
* fewer exceptions  
* faster decision-making

## **Value Rule**

Every workflow should eventually connect to a value hypothesis.

A value hypothesis says:

If we improve this loop, the business benefits in this way.

Examples:

* If follow-up is faster, revenue conversion improves.  
* If evidence is complete, compliance risk decreases.  
* If data is cleaner, reporting becomes more reliable.  
* If workflow is documented, dependency risk decreases.  
* If exceptions are escalated sooner, customer impact decreases.

---

# **12\. Feedback**

## **Definition**

Feedback is what the system learns after action is taken.

Feedback asks:

* Was the work completed?  
* Was it completed correctly?  
* Was the evidence sufficient?  
* Did the expected value occur?  
* Was the agent proposal accepted?  
* Was it rejected?  
* Did the human override the system?  
* Did the workflow create unnecessary friction?  
* Did the issue recur?  
* Should the rule change?  
* Should the configuration change?  
* Should this be automated next time?

## **Feedback Sources**

Feedback may come from:

* user actions  
* manager review  
* completion data  
* repeated exceptions  
* agent acceptance/rejection  
* KPI movement  
* human comments  
* customer outcomes  
* financial outcomes  
* audit/review outcomes

## **Feedback Rule**

Feedback must be captured as structured data where possible.

The system should not only know what happened.

It should know whether the loop worked.

---

# **13\. Adaptation**

## **Definition**

Adaptation is how the OS improves.

After feedback, the system determines whether to:

1. Repeat  
2. Deviate  
3. Innovate

## **Repeat**

Repeat when the loop worked.

The system should make successful loops easier, faster, and more automatic.

Repeat means:

* same trigger  
* same workflow  
* same owner/role  
* same evidence requirement  
* same report output  
* possibly more automation

## **Deviate**

Deviate when the loop mostly worked but needs adjustment.

Deviation may include:

* different owner  
* different checklist  
* different threshold  
* different escalation rule  
* different evidence requirement  
* different timing  
* different approval path  
* different agent prompt

## **Innovate**

Innovate when the existing loop is insufficient.

Innovation may include:

* new workflow  
* new metric  
* new input source  
* new automation key  
* new widget  
* new report  
* new service offering  
* new operating standard  
* new business outcome

## **Adaptation Rule**

The OS should be designed to improve loops over time.

The system should not simply store work.

It should help the business learn how it operates.

---

# **14\. ROI and Impact Layer**

## **Definition**

The ROI layer connects operating activity to measurable business improvement.

It should track both direct and indirect impact.

## **Direct ROI**

Direct ROI includes:

* revenue generated  
* cost avoided  
* hours saved  
* faster collections  
* improved conversion  
* reduced rework  
* reduced vendor spend  
* reduced admin load

## **Indirect ROI**

Indirect ROI includes:

* better data quality  
* less founder dependency  
* stronger compliance record  
* cleaner reporting  
* better staff accountability  
* improved customer experience  
* improved readiness for financing, sale, acquisition, or scaling

## **ROI Rule**

Not every work item needs a dollar value.

But every recurring workflow should have a value category.

The system should be able to show:

* what work was done  
* why it mattered  
* what changed  
* what value was likely created  
* whether the loop should repeat, deviate, or innovate

---

# **15\. The Business Mechanics Engine**

Dispatch OS is a business mechanics engine.

It should help answer:

* What is entering the business?  
* What does it mean?  
* What should happen?  
* Who should do it?  
* What proves it happened?  
* What value did it create?  
* What should happen next time?

The product is not the input.  
The product is not the agent.  
The product is not the dashboard.  
The product is the loop.

The loop turns business noise into operating memory and business improvement.

# **16\. Development Rules**

## **Rule 1: Preserve the Loop**

Every feature should map to at least one stage of the loop:

Input → Context → Interpretation → Decision → Action → Evidence → Value → Feedback → Adaptation

If a feature does not support the loop, question whether it belongs in the core.

## **Rule 2: Keep Core Generic**

Core primitives must be industry-agnostic.

Specific business meaning belongs in configuration.

## **Rule 3: Stage Before Truth**

Inputs and agent outputs should be staged before becoming operational truth.

## **Rule 4: Humans Approve Important Meaning**

Agents propose. Humans approve. Events preserve.

## **Rule 5: Evidence Is Required for Trust**

The system should preserve proof, not just status.

## **Rule 6: Metrics Must Connect to Work**

Dashboards should emerge from execution, evidence, and outcomes.

Do not build vanity dashboards.

## **Rule 7: Configuration Creates Specificity**

The same core should support different business types through configuration, not custom forks.

## **Rule 8: Feedback Improves the System**

Every repeated workflow should become smarter through usage.

## **Rule 9: Do Not Overbuild Integrations First**

Manual uploads, exports, and human-assisted ingestion are acceptable starting points.

The first goal is operating usefulness, not perfect automation.

## **Rule 10: The OS Must Make the Business More Legible**

Every implementation should make the business easier to understand, manage, measure, improve, finance, transfer, or scale.
