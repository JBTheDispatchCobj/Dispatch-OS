# **4\. SOURCE-HANDLING RULES**

## **Preserve Source Meaning**

Use the terminology, headings, order, and framing contained in the source material.

Do not silently replace source terminology with newer terminology.

## **Distinguish Extraction from Interpretation**

A segmented document may be a direct extraction from a larger concatenated source.

The extraction itself is not automatically canonical product architecture.

It is a source artifact awaiting classification.

## **Do Not Reconcile Conflicts Prematurely**

When two documents disagree:

* preserve both positions  
* identify the conflict  
* note the relevant source and generation  
* defer the canonical decision until the document-control review

## **Do Not Trust Historical Completion Labels Automatically**

Terms such as:

* production draft  
* complete  
* canonical  
* final  
* 90%  
* architecture complete

must be evaluated against the current V1 completion standard.

A historical file may call itself complete while lacking schemas, permissions, fixtures, tests, validation, or implementation artifacts.

## **Do Not Penalize Strong Narrative Work Merely Because It Is Old**

Older material may contain foundational thinking that remains stronger or broader than later FS sections.

Assess its actual substance.

# **7\. AUTHORITY STATUS MODEL**

Assign each artifact one primary authority status.

## **Canonical**

The artifact is currently the authoritative source of truth.

## **Canonical Input**

The artifact remains valid but has been incorporated into a later canonical specification.

## **Active Draft**

The artifact is current and useful but incomplete.

## **Historical Foundation**

The artifact explains original product thinking but is no longer the operating specification.

## **Superseded**

A later artifact replaces it.

## **Partially Superseded**

Some sections remain useful while other sections were replaced.

## **Reference Research**

The material informs the product but is not binding architecture.

## **Source Compilation**

The artifact combines multiple documents or thought streams and requires segmentation or classification.

## **Duplicate**

Its substantive content is captured elsewhere.

## **Unresolved**

Authority cannot yet be determined.

# **8\. MATURITY STANDARD**

Maturity should be evaluated using the current Dispatch V1 content-completion standard.

## **0–25% — Conceptual**

Contains:

* ideas  
* thesis  
* brainstorming  
* directional research  
* minimal operational structure

## **25–40% — Directional Architecture**

Contains:

* major components  
* broad relationships  
* some object or workflow concepts  
* substantial missing structure

## **40–60% — Substantive Draft**

Contains:

* meaningful product architecture  
* taxonomy  
* object lists  
* workflows or operating models  
* incomplete integration or governance detail

## **60–75% — Strong Baseline**

Contains:

* most product logic  
* major states  
* actors  
* objects  
* workflows  
* acceptance concepts  
* meaningful dependencies

But may lack:

* machine-readable schemas  
* exhaustive fields  
* permission matrices  
* API/event contracts  
* fixtures  
* tests  
* runbooks  
* validation

## **75–90% — Refined Implementation Specification**

Contains:

* detailed architecture  
* clear operational behavior  
* meaningful implementation artifacts  
* limited remaining gaps in tests, fixtures, reconciliation, or validation

## **90–99% — Release Candidate Documentation**

Contains nearly all required V1 content but still has:

* unresolved references  
* validation gaps  
* missing test execution  
* incomplete manifests  
* minor cross-section conflicts

## **100% — V1 Content-Complete**

Contains:

* complete narrative  
* schemas  
* field dictionaries  
* decision rules  
* permissions  
* approvals  
* evidence requirements  
* API/event contracts  
* fixtures  
* sample outputs  
* exception handling  
* threat model  
* tests  
* runbooks  
* release manifest  
* integrity index  
* passing validation

This means documentation-complete, not software-complete.

# **10\. SPECIAL HANDLING OF THE LARGE CONCATENATED SOURCE**

The large source file contains multiple embedded documents, frameworks, and volume-like sections.

Treat it as:

## **Source Compilation**

Do not assign one maturity percentage to the entire file.

Instead:

1. Identify each internal document boundary.  
2. Create an artifact record for each internal document.  
3. Preserve the original heading.  
4. Preserve the original order.  
5. Record the source location.  
6. Assign maturity independently.  
7. Map each segment independently.  
8. Identify repeated or conflicting sections.

A large file may contain:

* highly developed ontology material  
* shallow lists  
* completed architecture  
* speculative ideas  
* obsolete assumptions  
* duplicated frameworks

A single percentage would be misleading.
