# FS-15006 — Named Terminal Product Model
## The Institution-Branded External Product Powered by Dispatch OS

**Section:** FS-15000 — The Auric, Named Terminal & Institutional Network Flywheel  
**Document:** 15006  
**Status:** 75% Product Doctrine Baseline  
**Authority:** Controlling doctrine for the market-facing product model of all institution-, company-, partner-, and network-branded Terminals  
**Applies to:** Named Terminals, white-label configuration, tenant identity, public and private profile layers, shared network services, institutional workspaces, private cartridges, partner deployments, branding, permissions, governance, workflows, and commercialization

---

# 1. Purpose

This document defines the Named Terminal as Auric Works’ primary external product.

A Named Terminal is the branded institutional operating environment through which an institution, company, CUSO, partner, or network:

- understands itself
- receives relevant intelligence
- manages profiles
- detects opportunities
- evaluates readiness
- coordinates relationships
- launches governed engagement
- manages evidence and approvals
- retains institutional memory
- interacts with the broader market

The customer-facing product is not required to be called “Dispatch Terminal.”

The product should ordinarily be named for the institution, company, partner, or network operating it.

Examples:

- Alloya Terminal
- NACUSO Terminal
- Heartland Credit Union Terminal
- Prairie Community Bank Terminal
- Example Fintech Terminal
- Cooperative Markets Terminal
- Partner Network Terminal

Dispatch OS remains the proprietary operating architecture beneath every Named Terminal.

The controlling doctrine is:

**The Named Terminal is the external product.  
Dispatch OS is the operating architecture.  
The Auric is the public intelligence layer.  
Cartridges, Studio, Developer, Enterprise, and Partner capabilities configure and extend the Terminal rather than fragmenting it into separate products.**

---

# 2. Why the Named Terminal Model Exists

Financial institutions and market participants do not want to assemble an AI operating stack.

They do not want to manage:

- model providers
- agent frameworks
- knowledge graphs
- workflow engines
- evidence systems
- approval systems
- connector runtimes
- profile infrastructure
- readiness frameworks
- matching engines
- editorial feeds
- network infrastructure

They want an operating environment that is already structured around their institution, market, users, authority, systems, risks, and goals.

The Named Terminal model allows Auric Works to deliver common infrastructure through a customer-specific experience.

It creates:

- institutional ownership of the front-end experience
- Auric Works ownership of the operating architecture
- shared network intelligence
- private institutional context
- reusable cartridges
- configurable workflows
- white-label credibility
- centralized product governance
- lower customer complexity
- scalable deployment

The model also supports partners that want to serve a market or membership without building the full product themselves.

# 29. Canonical Implementation Model

## 29.1 Canonical Objects

The Named Terminal product model SHALL be implemented through the following canonical objects, all registered through FS-5100 and linked to FS-6000 tenant, institution, authority, evidence, approval, and audit objects:

| Object | Purpose | Canonical owner | Required identity |
|---|---|---|---|
| `named_terminal` | Top-level externally branded product instance | Auric Works platform | `terminal_id` |
| `terminal_tenant_binding` | Binds a Terminal to one or more governed tenants | Platform administration | `binding_id` |
| `terminal_brand_configuration` | Stores approved customer-facing identity | Tenant brand owner | `brand_config_id` |
| `terminal_domain_binding` | Binds approved domains and routing rules | Platform security | `domain_binding_id` |
| `terminal_class_assignment` | Declares institution, company, network, partner, or public class | Product governance | `class_assignment_id` |
| `terminal_capability_assignment` | Enables a registered capability | Tenant configuration owner | `capability_assignment_id` |
| `terminal_cartridge_assignment` | Enables and versions an FS-5100 cartridge | Cartridge administrator | `cartridge_assignment_id` |
| `terminal_workspace_assignment` | Maps personas and roles to FS-10000 workspaces | Tenant administrator | `workspace_assignment_id` |
| `terminal_data_boundary` | Defines public, claimed, private, and governed-shared scope | Data owner | `data_boundary_id` |
| `terminal_network_participation` | Governs participation in shared network services | Executive approver | `network_participation_id` |
| `terminal_partner_delegation` | Grants bounded administration to a partner | Tenant authority | `delegation_id` |
| `terminal_configuration_change` | Immutable proposed and approved configuration delta | Change owner | `change_id` |
| `terminal_release_binding` | Pins the Terminal to approved platform and cartridge versions | Release manager | `release_binding_id` |
| `terminal_export_request` | Governs portability and termination exports | Data owner | `export_request_id` |
| `terminal_suspension` | Records partial or complete operational suspension | Security or platform owner | `suspension_id` |
| `terminal_service_entitlement` | Records commercially entitled services and limits | Commercial operations | `entitlement_id` |

## 29.2 Canonical Identity and Namespace Rules

1. Every Terminal SHALL have one immutable `terminal_id` and one human-readable `terminal_slug`.
2. A `terminal_slug` SHALL be unique within the global platform namespace.
3. Display names may change without changing `terminal_id`.
4. Customer-owned domains SHALL be verified before activation.
5. Every Terminal SHALL bind to at least one tenant and exactly one active Terminal class.
6. A Network Terminal may bind to multiple member tenants only through explicit network-participation objects.
7. Partner access SHALL never imply tenant ownership.
8. Public profile identity SHALL remain separate from private tenant identity even when both represent the same institution.
9. Historical names, prior domains, and aliases SHALL be retained as noncanonical aliases.
10. Terminal deletion SHALL be represented as termination and archival, not destructive identity reuse.

# 33. Lifecycle State Machine

## 33.1 Terminal States

`draft → provisioning → validation → active → restricted → suspended → terminating → archived`

Permitted reactivation paths:

- `restricted → active` after corrective validation.
- `suspended → validation` after security or contractual remediation.
- `terminating → active` only before the irreversible termination checkpoint and with executive, legal, and security approval.

|---|---|---|---|---|---|

| active | Material defect detected | Severity threshold met | Security owner for planned restriction; automatic emergency restriction allowed with immediate review | restricted | Incident or control finding |
| restricted | Remediation passed | Failed controls corrected | Security owner | active | Closure evidence |

| active/restricted | Termination notice accepted | Contract and export plan approved | Commercial, legal, tenant authority | terminating | Termination plan |

# 38. Public and Private Profile Synchronization Rules

1. Public and claimed fields SHALL never overwrite private tenant fields.
2. Private fields MAY propose updates to public or claimed fields but require verification and publication approval.
3. Public-record updates SHALL create a comparison event, not an automatic private overwrite.
4. Claimant corrections SHALL retain prior values and evidence lineage.
5. A private field marked confidential SHALL never be used as public-profile evidence.
6. A governed-shared outcome MAY update public history only if the sharing agreement and involved parties permit publication.
7. Stale public data SHALL remain visible as stale until refreshed, corrected, or superseded.
8. Conflicts between public, claimed, attested, and inferred values SHALL create a discrepancy object.
9. Agent-generated profile suggestions SHALL remain drafts until human verification.
10. Network matching SHALL use only data classes authorized for the matching purpose.
