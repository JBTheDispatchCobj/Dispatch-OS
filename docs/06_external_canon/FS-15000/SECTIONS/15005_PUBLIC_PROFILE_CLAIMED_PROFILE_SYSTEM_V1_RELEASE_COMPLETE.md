**Section:** FS-15000 — The Auric, Named Terminal & Institutional Network Flywheel

**Status:** 75% Product Doctrine Baseline  
**Authority:** Controlling doctrine for public profiles, claimed profiles, verification, disputes, private enrichment, readiness, eligibility, and profile-driven acquisition  
**Applies to:** Institutions, credit unions, community banks, CUSOs, fintechs, vendors, executives, boards, products, systems, partners, investors, regulators, public profiles, private profile overlays, claim workflows, evidence, matching, and engagement

---

# 1. Purpose

This document defines the profile system that connects The Auric’s public institutional record to the private operating environment inside each Named Terminal.

The profile system is the bridge between:

- public intelligence
- institutional identity
- company identity
- market discoverability
- evidence
- readiness
- matching
- private operating context
- governed engagement
- verified reputation

The system must support two distinct but connected profile layers:

1. **Public Profile** — the evidence-backed public record maintained through The Auric and Dispatch-powered institutional intelligence.
2. **Claimed Profile** — the verified, participant-managed extension of that record, including private information, readiness, evidence, engagement preferences, and institution-specific context.

The governing doctrine is:

**The public profile establishes what the market can responsibly know.  
The claimed profile establishes what the participant can verify, enrich, and operate.  
The private profile overlay establishes what the participant can use inside a Named Terminal without exposing it publicly.**

Claiming a profile must create real operating value.

It must not be a vanity action, a directory listing, or a paid badge.

# 5. The Claimed Profile

A participant may claim a profile after identity verification.

The claimed profile allows the participant to:

- verify official identity
- correct basic information
- add company-supplied information
- add product information
- add executives
- add implementation context
- upload evidence
- define use cases
- map systems and integrations
- establish engagement preferences
- complete readiness
- manage authorized public references
- manage private institutional information
- receive opportunities
- participate in governed workflows

Claiming does not transfer editorial ownership of the public profile.

The claimant may:

- propose corrections
- submit evidence
- supplement information
- dispute claims
- verify fields
- manage private fields

The claimant may not:

- remove accurate, properly sourced public history
- convert marketing claims into verified facts
- suppress editorial coverage
- alter public rankings without methodology-based review
- use payment to obtain favorable verification

# 8. Verification States

Profile fields should carry explicit verification states.

## 8.1 Publicly Sourced

Supported by an external source.

## 8.2 Claimant Submitted

Submitted by the profile owner but not independently verified.

## 8.3 Claimant Verified

Verified through claimant authority and evidence.

## 8.4 Independently Verified

Confirmed through authoritative external evidence.

## 8.5 Attested

Confirmed by an authorized individual or institution.

## 8.6 Inferred

Derived from multiple signals.

## 8.7 Disputed

Challenged by a participant.

## 8.8 Stale

No longer within freshness threshold.

## 8.9 Superseded

Replaced by a newer field value while preserving history.

## 8.10 Unknown

No reliable value exists.

The system must not collapse these states into a single “verified” badge.

# 26. Controlling Statement

The profile system is the identity spine of the external product.

The Auric maintains the public record.

Participants claim, verify, and enrich their identity.

Named Terminals add private operating context, readiness, evidence, and engagement capability.

Dispatch OS preserves the distinction among public fact, claimant submission, private information, inference, dispute, and verified outcome.

The system wins when profiles become more accurate, more useful, and more valuable with every interaction—without allowing money, ownership, or pressure to rewrite institutional truth.

# 27. Canonical Implementation Model

This release-complete specification preserves the doctrine above and makes the profile system executable as a governed product capability.

The canonical aggregate is `profile_record`. Each profile record contains one public projection, zero or more verified claim relationships, zero or more private tenant overlays, a field-level provenance ledger, dispute history, completeness/readiness calculations, and relationship/event links.

## 27.1 Canonical Objects

| Object ID | Object | Purpose | Canonical Authority |
|---|---|---|---|
| `OBJ-PROFILE-RECORD` | Profile Record | Root identity aggregate | FS-15005 / FS-5100 registry |
| `OBJ-PROFILE-PROJECTION` | Profile Projection | Public, claimed, shared, or private view | FS-15005 |
| `OBJ-PROFILE-FIELD` | Profile Field | Field value plus provenance, verification, freshness, and visibility | FS-15005 |
| `OBJ-PROFILE-CLAIM` | Profile Claim | Request to administer a profile | FS-15005 |
| `OBJ-CLAIMANT-AUTHORITY` | Claimant Authority | Evidence that a user may act for the subject | FS-6000 authority model |
| `OBJ-PROFILE-EVIDENCE-LINK` | Profile Evidence Link | Connects a field or claim to evidence | FS-6000 evidence model |
| `OBJ-PROFILE-DISPUTE` | Profile Dispute | Governed challenge to a field or record | FS-15005 |
| `OBJ-PROFILE-CORRECTION` | Profile Correction | Proposed and resolved amendment | FS-15005 |
| `OBJ-PROFILE-OVERLAY` | Private Profile Overlay | Tenant-private enrichment | FS-15005 / FS-12000 |
| `OBJ-PROFILE-SHARE-GRANT` | Share Grant | Time-bound, purpose-bound private disclosure | FS-12000 |
| `OBJ-PROFILE-COMPLETENESS` | Completeness Assessment | Use-case-specific profile coverage | FS-15005 |
| `OBJ-PROFILE-READINESS` | Readiness Assessment | Evidence-backed readiness state | FS-6000 readiness model |
| `OBJ-VERIFIED-OUTCOME` | Verified Outcome | Outcome that may affect reputation | FS-15005 |

## 27.2 Identity Keys

Every profile must preserve:

- `profile_id`: immutable Dispatch identifier.
- `subject_type`: institution, company, executive, board, committee, product, relationship, or market theme.
- `canonical_subject_id`: FS-6000/FS-9000 subject identifier where available.
- `legal_identity_key`: charter, registration, tax, LEI, NCUA charter, state registration, or equivalent identifier where lawful.
- `display_identity_key`: normalized public name plus jurisdiction and type.
- `tenant_overlay_key`: tenant plus profile identifier.
- `version`: monotonically increasing integer.
- `effective_at`, `observed_at`, and `recorded_at` timestamps.

No name string alone is sufficient to establish identity.

# 30. Claim Lifecycle State Machine

```text
UNCLAIMED
  -> CLAIM_REQUESTED
  -> IDENTITY_VERIFICATION
  -> AUTHORITY_VERIFICATION
  -> SCOPE_REVIEW
  -> RECORD_REVIEW
  -> EVIDENCE_REVIEW
  -> APPROVED
  -> ACTIVE

ACTIVE -> REVERIFICATION_DUE -> ACTIVE
ACTIVE -> SUSPENDED -> ACTIVE | REVOKED
ACTIVE -> EXPIRED -> CLAIM_REQUESTED
ANY_NONFINAL -> REJECTED
ANY_STATE -> FRAUD_HOLD -> SUSPENDED | REVOKED | ACTIVE
```

## 30.1 Transition Rules

| From | To | Guard | Required Evidence | Approver |
|---|---|---|---|---|
| Unclaimed | Claim requested | Authenticated user | Identity assertion | System |
| Claim requested | Identity verification | Request complete | Email/domain or identity proof | System + reviewer for exceptions |
| Identity verification | Authority verification | Identity passed | Employment, officer, delegated authority, board authorization, or equivalent | Profile claim reviewer |
| Authority verification | Scope review | Authority passed | Requested scope | Profile steward |
| Scope review | Record review | Scope permitted | Existing public record | Claimant + steward |
| Record review | Evidence review | Proposed changes submitted | Field-level evidence | Evidence reviewer |
| Evidence review | Approved | All material requirements passed | Approval record | Authorized human reviewer |
| Active | Suspended | Authority lapse, fraud signal, dispute, or policy breach | Investigation record | Governance reviewer |
| Suspended | Revoked | Material failure confirmed | Closure evidence | Governance authority |

No agent may approve a claim, restore a suspended claim, or resolve a material dispute without human approval.

# 33. Completeness Scoring Model

Completeness is use-case-specific and must never be presented as a universal quality score.

## 33.1 Formula

For use case `u`:

`completeness(u) = sum(weight_i * coverage_i * freshness_i * verification_i) / sum(weight_i)`

Where:

- `coverage_i` is 1 when present, 0 otherwise, or partial for structured subfields.
- `freshness_i` is 1 current, 0.75 aging, 0.25 stale, 0 unknown.
- `verification_i` is 1 independently verified, 0.9 claimant verified, 0.8 attested, 0.6 publicly sourced, 0.4 claimant submitted, 0.25 inferred, 0 disputed/unknown.
- fields classified private are excluded from public completeness unless the use case explicitly permits them.

## 33.2 Completeness Bands

| Score | Band | Meaning |
|---:|---|---|
| 0–39 | Insufficient | Too little reliable information for the use case |
| 40–59 | Partial | Some useful information; material gaps remain |
| 60–74 | Operational baseline | Usable with visible gaps |
| 75–89 | Strong | Most required information is present and current |
| 90–100 | Verified complete | Nearly all required information is current and verified |

Completeness must always expose missing, stale, disputed, and inferred components.

# 34. Readiness Integration Contract

Profile data may feed FS-6000 readiness assessments only when:

1. The contributing field is mapped to an approved readiness input.
2. Source and verification state are preserved.
3. Private fields remain tenant-scoped.
4. The readiness model records the contributing profile field IDs.
5. A score does not silently convert unknown into zero.
6. A score change emits `profile.readiness.changed` and the FS-6000 readiness event.
7. Material readiness decisions remain reviewable and explainable.

# 37. Validation Rules

1. A public field cannot cite a private-only source without a publishable derivative source.
2. A claimed profile cannot exist without a valid claim record.
3. A claim cannot become active without identity and authority verification.
4. A private overlay must include tenant ID and classification.
5. Shared fields require an active, unexpired share grant.
6. Disputed values cannot be represented as independently verified.
7. Stale fields must remain explicitly stale until refreshed or superseded.
8. Unknown must remain distinct from false, empty, and zero.
9. Agent-generated content must remain labeled until accepted by an authorized human or workflow.
10. Duplicate legal identity keys must trigger entity-resolution review.
11. Profile merges must preserve both source histories and redirect references.
12. Verification changes require evidence or reviewer action.
13. Reputation impact requires a verified outcome, not a claim or payment.

# 42. Profile Operations Runbook

## 42.1 Daily

- review failed ingestion and entity-resolution queues
- review claim and dispute SLA breaches
- monitor unauthorized access and share-grant anomalies
- inspect stale critical fields
- verify public/private projection isolation

## 42.2 Weekly

- review fraud holds and contested claims
- sample published changes for evidence quality
- review duplicate-profile queue
- review claim abandonment and conversion
- audit agent-generated changes awaiting review

## 42.3 Monthly

- rotate evidence samples for independent QA
- review completeness weights and use-case definitions
- validate role and authority mappings
- review commercial/editorial separation
- reconcile profile IDs with FS-5100 and FS-9000

## 42.4 Incident Procedure

1. Contain affected profile, claim, field, or share grant.
2. Preserve audit and evidence records.
3. Classify privacy, integrity, fraud, or authority impact.
4. Notify profile steward, security, and affected parties as required.
5. Correct projection and access state.
6. Record closure evidence and root cause.
7. Add prevention control or test.

# 43. Migration and Deduplication Runbook

1. Inventory legacy profile sources and identifiers.
2. Assign source type, authority, and freshness.
3. Resolve legal identity keys before names.
4. Create candidate clusters using FS-9000 entity resolution.
5. Require human review for ambiguous or high-impact merges.
6. Preserve source lineage and prior identifiers.
7. Create redirect aliases for merged profiles.
8. Migrate public and private fields into separate projections.
9. Mark unknown, stale, disputed, and inferred states explicitly.
10. Validate tenant isolation before enabling overlays.
11. Recalculate completeness only after migration validation.
12. Publish migration report and rollback checkpoint.
