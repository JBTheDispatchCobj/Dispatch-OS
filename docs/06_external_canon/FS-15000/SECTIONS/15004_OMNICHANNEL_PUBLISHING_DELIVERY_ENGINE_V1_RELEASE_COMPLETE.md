# 1. Purpose

This document defines the omnichannel publishing and delivery architecture for The Auric and the Named Terminal network.

The system must allow one canonical content or intelligence object to produce multiple governed channel-specific expressions without losing:

- factual consistency
- source lineage
- disclosure
- version control
- audience relevance
- correction state
- permissions
- institutional context
- commercial separation
- performance attribution

The objective is not simply to post the same article everywhere.

The objective is to transform one authoritative intelligence object into the right format, length, tone, depth, timing, and level of context for each audience and channel.

The governing doctrine is:

**Create once at the canonical level. Adapt deliberately by channel. Deliver according to institutional relevance. Measure the resulting intelligence and action. Correct from the source of truth.**

# 3. Governing Principles

## 3.1 Canonical First

No channel should become an independent source of truth.

Every derivative should point back to a canonical content or intelligence object.

## 3.2 Audience Relevance Before Volume

Distribution should be based on institutional and role relevance, not only audience size.

## 3.3 Channel-Native Expression

Each channel should receive content designed for its constraints and user behavior.

## 3.4 Disclosure Preservation

Sponsored, submitted, editorial, opinion, and commercial distinctions must survive transformation.

## 3.5 Correction Propagation

Corrections should flow from the canonical object to every active derivative where operationally possible.

## 3.6 Permission Awareness

Private institutional context must never leak into public distribution.

## 3.7 Human Control of Material Meaning

Automation may adapt and format, but may not materially change factual meaning, disclosure, risk framing, or editorial conclusion without review.

## 3.8 Delivery Is Part of Product Logic

Distribution should improve profiles, feeds, relationships, workflows, and institutional intelligence.

# 27. V1 Release-Complete Implementation Specification

This section converts the governing doctrine above into the production-ready implementation contract for the omnichannel publishing and delivery engine. It preserves the canonical-first model and adds the schemas, registries, decision rules, permissions, APIs, events, fixtures, tests, runbooks, and release controls required for implementation.

## 27.1 Identifier and Namespace Conventions

Canonical identifiers use stable, tenant-aware prefixes:

- `CHAN-` channel definition
- `AUD-` audience segment
- `DELIVRULE-` delivery rule
- `DERIV-` derivative content object
- `DELIVERY-` delivery instance
- `SCHED-` schedule
- `NEWS-` newsletter manifest
- `FEEDITEM-` Named Terminal feed item
- `SYND-` syndication agreement or delivery
- `CORRPROP-` correction propagation job
- `PREF-` delivery preference
- `ANEVT-` analytics event

Identifiers are immutable. Display names may change. All objects include `tenant_id`, `version`, `status`, `created_at`, `updated_at`, `created_by`, and `source_system` where applicable.

## 27.6 Delivery Instance and State Machine

Delivery states:

`created → eligibility_checked → permission_checked → transformed → review_pending → approved → scheduled → queued → sending → delivered → acknowledged`

Exception branches:

- `eligibility_failed`
- `permission_denied`
- `transformation_failed`
- `review_rejected`
- `expired`
- `send_failed`
- `partially_delivered`
- `suppressed`
- `cancelled`
- `recalled`
- `correction_pending`
- `corrected`

Each transition records actor, timestamp, input version, output version, reason code, evidence, and retry count.

## 27.7 Institutional Relevance Scoring Model

The default V1 relevance score is explainable and bounded from 0 to 100:

| Factor | Weight | Example signal |
|---|---:|---|
| Institution match | 20 | institution explicitly named or affected |
| Role relevance | 15 | topic maps to executive responsibility |
| Product/vendor/system match | 15 | institution uses affected technology |
| Regulatory applicability | 15 | rule applies by charter, asset size, geography, or activity |
| Strategic-priority match | 10 | aligns to current institutional priority |
| Relationship proximity | 10 | known partner, vendor, peer, or executive relationship |
| Workflow relevance | 10 | active workflow or unresolved task is affected |
| Freshness/materiality | 5 | recent and operationally material |

Rules:

1. Missing data does not score as a negative unless explicitly defined.
2. Inferred facts contribute at reduced confidence.
3. Private signals may rank private Terminal delivery but may not appear in public output.
4. Regulatory or safety-critical events may override ordinary ranking thresholds.
5. The user can inspect the major factors that caused delivery.

|---|---|---|---|---|
| Published article | Public web | canonical article page | existing editorial approval | changing conclusion |
