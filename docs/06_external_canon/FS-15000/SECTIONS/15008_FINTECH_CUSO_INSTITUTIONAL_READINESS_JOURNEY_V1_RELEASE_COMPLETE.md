# 1. Purpose

This document defines the end-to-end institutional readiness journey for fintechs, CUSOs, vendors, and related companies seeking to engage credit unions, community banks, and other financial institutions through the Named Terminal network.

The purpose of the journey is to replace fragmented market entry with a governed path from:

- discovery
- public profile
- claimed identity
- institutional package
- evidence
- readiness
- eligibility
- matching
- engagement
- pilot
- implementation
- verified outcome
- compounding reputation

The governing doctrine is:

**A company should not join the network merely for exposure. It should join because the system helps it become institution-ready, understand what blocks execution, improve the quality of its institutional package, reach relevant institutions, and convert successful work into durable credibility.**

The journey must create value before the company has a customer, during engagement, and after successful implementation.

---

# 2. The Market Failure

Fintechs, CUSOs, and vendors often struggle to engage institutions because the market is structurally inefficient.

Companies may have:

- strong products
- credible founders
- technical capability
- customer interest
- capital
- early references

Yet still fail to enter institutions because they lack:

- institution-specific use cases
- complete evidence
- clear data flows
- integration plans
- implementation plans
- pricing clarity
- security readiness
- compliance readiness
- executive sponsorship
- referenceability
- understanding of institutional authority
- understanding of vendor-risk processes
- understanding of board or committee expectations

Institutions, in turn, struggle to determine:

- what the company actually does
- whether the use case is real
- whether the company is ready
- whether the institution is ready
- what evidence is missing
- what systems are affected
- who must approve
- whether the company can implement
- whether a pilot can be controlled
- whether the outcome can be measured

The readiness journey should make both sides legible before engagement begins.

## 33. Readiness Scoring Model

Each readiness dimension is scored from 0–100 but must never be presented without evidence coverage and confidence.

`dimension_score = weighted_control_score × evidence_coverage × freshness_factor × confidence_factor`

Where:

- `weighted_control_score` = weighted percentage of satisfied requirements
- `evidence_coverage` = evidenced requirements / required requirements
- `freshness_factor` = 1.00 current, 0.85 aging, 0.60 stale, 0.00 expired where blocking
- `confidence_factor` = 1.00 verified, 0.85 attested, 0.60 inferred, 0.00 unknown where blocking

### 33.1 V1 Dimension Weights

| Dimension | Default Weight |
|---|---:|
| Company | 10% |
| Product | 12% |
| Security | 15% |
| Compliance | 13% |
| Integration | 12% |
| Data | 10% |
| Implementation | 10% |
| Institutional Fit | 8% |
| Pilot | 5% |
| Commercial | 5% |

Weights may vary by engagement type but changes require a versioned scoring policy.

### 33.2 Blocking Rules

A high aggregate score may not override a material blocker. Blocking categories include:

- identity not verified
- missing authority to submit evidence
- prohibited data use
- unresolved critical security issue
- absent incident-response capability where required
- inability to meet mandatory regulatory obligation
- material misrepresentation
- no implementation owner
- no human approval authority
- unresolved conflict of interest

## 35. Workflow Definitions

### 35.1 Claim and Verification Workflow

Trigger: authorized company representative initiates claim.

Steps:
1. Capture claimant identity.
2. Verify official domain and corporate relationship.
3. Verify legal entity and authority.
4. Detect competing claims.
5. Request additional evidence when confidence is insufficient.
6. Human reviewer approves, rejects, or conditionally approves.
7. Emit claim event and update profile permissions.

### 35.2 Readiness Assessment Workflow

Trigger: claimed company submits a product/use-case package.

Steps:
1. Freeze package version.
2. Resolve required requirements by product, institution class, engagement type, geography, and data classification.
3. Validate evidence presence, type, owner, freshness, and contradictions.
4. Run deterministic scoring rules.
5. Produce gaps and blockers.
6. Assign remediation owners.
7. Human reviewer confirms material findings.
8. Publish controlled readiness result.

### 35.3 Governed Introduction Workflow

Trigger: eligibility and bilateral fit are satisfied.

Steps:
1. Confirm preferences and exclusions.
2. Confirm institution sponsor.
3. Run conflict and restricted-party checks.
4. Prepare bounded opportunity summary.
5. Obtain network-operator approval.
6. Obtain receiving-institution acceptance.
7. Open controlled engagement workspace.
8. Preserve introduction provenance and consent.

### 35.4 Pilot-to-Outcome Workflow

Trigger: approved pilot charter.

Steps:
1. Validate prerequisites and approvals.
2. Provision limited environment and data access.
3. Monitor milestones, controls, incidents, and KPIs.
4. Record deviations and remediation.
5. Close or terminate pilot.
6. Verify outcome against charter metrics.
7. Obtain both-side attestations.
8. Publish only permitted reputation signals.

- identity cannot become verified without reviewer approval

- expired evidence changes affected readiness state
- conditional eligibility carries enforceable conditions and expiration
- pilot cannot start without required approvals and charter
- verified outcome requires source evidence and authorized attestations
- disputed outcome does not improve reputation
- all events validate against event schemas

## 45. Migration Guidance

Existing company or vendor records must be migrated without inventing readiness.

Migration sequence:

1. Import source identity and provenance.
2. Deduplicate against canonical organization records.
3. Mark unclaimed profiles as public/provisional.
4. Separate marketing claims from verified facts.
5. Import evidence with original dates and classifications.
6. Set unknown fields explicitly.
7. Create readiness assessments only after applicable requirement sets are resolved.
8. Require new human approval before granting eligibility.

Legacy scores may be retained as historical but must not be treated as active V1 readiness decisions.
