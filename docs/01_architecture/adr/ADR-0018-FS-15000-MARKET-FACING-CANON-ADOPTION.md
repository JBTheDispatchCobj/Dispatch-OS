# ADR-0018 — Adopt the FS-15000 "Market-Facing Institutional Network V1" package as reference/operational canon (identity-reconciled, subordinate to the Dispatch Constitution)

- **Status:** Accepted (2026-07-22)
- **Deciders:** Bryan
- **Context sources:** the `FS-15000_MARKET_FACING_INSTITUTIONAL_NETWORK_V1_RELEASE_COMPLETE` package (15 sections FS-15001…FS-15015, doc-validated; `VALIDATION_REPORT.json` = "documentation-complete validation only. Software execution, live integrations, penetration testing, and production deployment were not performed"); ADR-0017 (FS V1 canon adoption); ADR-0008 (Constitution V1 adoption); `CLAUDE.md` specification hierarchy + change policy.
- **Relationship to ADR-0017:** ADR-0017 adopted the operating-system canon (FS-4000…FS-14000) as reference/operational canon and explicitly anticipated pulling in the downstream sections "as their waves arrive." **FS-15000 is that downstream family — the market-facing layer — and this ADR adopts it on the same terms.**

## Context

A second externally-authored (ChatGPT-generated), documentation-complete V1 specification arrived: **FS-15000 — the market-facing institutional network.** Where FS-4000…FS-14000 spec the operating system itself (objects, kernel, runtime, connectors, security), FS-15000 specs **how the outside world experiences it**: the Auric as publication-of-record, Named Terminals ("[Institution] Terminal"), the public/claimed/verified profile system, institutional readiness journeys, governed engagement, the relationship graph, matching + opportunity distribution, editorial integrity + commercial separation, the network flywheel, and a Product Constitution & Platform Charter.

Two facts shape the decision — the same two that shaped ADR-0017:

1. **It is the same product, fuller — and it *aligns* with what the repo has already built and tested.** A section-by-section review (all 15) found doctrinal ALIGNMENT with the running system and **zero contract conflicts**: the FS brand hierarchy (Auric / Named Terminal / Dispatch OS / cartridge) is the repo brand hierarchy verbatim; the FS provenance classes `{public, private, synthetic, attested, inferred, agent_generated, verified}` + data states `{current, aging/stale, unknown, disputed, superseded, restricted, pending-review}` map ~1:1 onto the repo truth planes (`shared_market` vs `private_terminal`) and surface states (`current/missing/stale/inferred/synthetic/restricted/pending_approval/conflicted`); "agents cannot publish / cannot approve," "matching never replaces a human institutional decision," "editorial truth is not for sale," "claiming ≠ verifying," "payment ≠ verification," and "ownership of a relationship = stewardship, not transaction authority" are the repo's human-gate + editorial/commercial-separation + first-class-relationship doctrine. It is the strongest downstream blueprint the repo has for the customer-facing sections it has only scaffolded.

2. **One section — FS-15015 — asserts competing top-of-stack authority.** It is headed *"Authority: Constitutional / controlling product doctrine,"* declares its principles *"immutable,"* defines its own §29 amendment procedure, and in §3 lays out a Constitutional Hierarchy that **omits the Dispatch Constitution (Vol I) entirely** and self-ranks at position #2 (below only law). Its *substance* is fully consistent with Vol I; only its *authority claim* collides with the repo's `CLAUDE.md` order (Constitution Vol I → ADRs → subsystem volumes → registries → `ACTIVE_BUILD.md`).

Treating the package as a competing build canon would invert the repo's build-the-running-slice discipline (the package's own README tells the reader to expand every section "to 100%" with schemas/registries/APIs — that is a documentation plan, not a build order). Ignoring it would waste a strong, aligned blueprint for exactly the market-facing surfaces the repo is now building (Sprint IV).

## Decision

1. **Adopt FS-15000 as REFERENCE / OPERATIONAL canon, not a build canon** — identical terms to ADR-0017. The repo's Constitution (Vol I) → ADRs → subsystem volumes → registries remain the authority for what is **built**. FS-15000 is the authority for **interpretation and verticalization** of the market-facing layer, and the source spec for those surfaces when their waves arrive. Stored at `docs/06_external_canon/FS-15000/` (sections under `SECTIONS/`, control files under `CONTROL/`), companion narrative `FS-15000_CANON_INTEGRATION.md`.

2. **Disposition the FS-15015 authority conflict explicitly (naming/precedence, not doctrine).** FS-15015 is adopted as an **operational product charter SUBORDINATE to the Dispatch Constitution (Vol I)**. Specifically:
   - Its self-declared *"Constitutional / controlling"* status, its *"immutable"* clause, and its §3 Constitutional Hierarchy are **reference-scoped** — they describe the market-facing product's internal logic; they do **not** bind or reorder the repo's build authority. Where FS-15015 §3 and `CLAUDE.md`'s authority order differ, **`CLAUDE.md` (Constitution Vol I first) governs.**
   - **Amendments to the market-facing charter flow through the repo ADR process**, not FS-15015 §29. A change to product doctrine is proposed as an ADR against Vol I, exactly as ADR-0008/0017 established — never as a silent charter amendment.
   - This is a label/precedence disposition only: the *substance* of FS-15015 (one-product rule, public/private separation, evidence-before-assertion, human authority, editorial-truth-not-for-sale) is consistent with Vol I and is adopted as operational doctrine.

3. **Wave order leads; sections are pulled in per wave as reference.** The package's "document every section to 100% first" sequence does **not** drive the build order. The next wave (Wave 4) pulls in **FS-15005 (Public/Claimed/Verified Profile System)** and **FS-15011 (Institutional Network & Relationship System)** because they are the spec for the repo's biggest current gap — the `/relationships` + `/executives` scaffolds need a `Relationship`/`PersonalProfile` store table + seed, and 15005/15011 define exactly the objects, state axes, and type registries such a store would carry (see the crosswalk). These are additive contracts; they amend no tested contract.

4. **Runtime identity reconciliation is DEFERRED until FS-15000 ships machine-readable registries.** Unlike FS-5100 (which shipped real registries and seeded `canon_aliases.json` under ADR-0017), FS-15000 is product *architecture* (~75%); its object/type registries (`REL-`, `RELTYPE-*`, profile subject types, engagement classes, edition/entitlement ids, conflict classes) are documentary, not contract-grade. **No new `canon_aliases.json` entries are added by this ADR.** When a Wave builds a live registry against an FS-15000 section, its identifiers are reconciled the ADR-0017 way — proposed by deterministic similarity, never auto-merged, human-confirmed, closed-graph-validated, gated by the debug-loop CANON step.

5. **New surfaces are noted, not created.** FS-15000 implies market-facing surfaces the 23-surface registry does not yet carry: a **public profile / claimed-profile** surface + claim workspace (15005), a **Named-Terminal editions / entitlement** admin surface (15006/15007), and the **public Auric publication** reader surface (15002/15004; partially covered by `/distribution`). These are recorded in the crosswalk as candidate registry entries for future waves; **none is added to `ui_surfaces.json` by this ADR** (surfaces are added when built, per the Wave-2/3 promote discipline).

6. **Arm's length is the standing rule.** Per the operator's direction and ADR-0017: the canon is a map you consult, not a to-do list you obey. A section is integrated only when a wave needs it and it passes the repo's own gate (contracts + tests + debug-loop). Its "constitutional/immutable" tone never overrides the running-slice discipline.

## Consequences

- The market-facing layer now has an adopted, aligned blueprint that the Sprint-IV Terminal work can build against, section by section, without re-deriving product doctrine — while the repo Constitution + gate discipline stay the authority for what actually ships.
- The FS-15015 authority ambiguity is closed: there is one build constitution (Vol I); FS-15015 is subordinate operational doctrine amended through ADRs. No "which constitution is real" drift.
- **No code, contract, migration, registry, or `ui_surfaces.json` entry changed by this ADR** — it is additive reference documentation plus this decision record. The debug-loop gate is unaffected (confirmed green at adoption).
- Follow-on (fan-out friendly, per wave): Wave 4 builds a `Relationship`/`PersonalProfile` store + seed from FS-15005/15011 and promotes `/relationships` + `/executives`; later waves pull FS-15006/15007 (Named-Terminal editions → `/administration` neighborhood), FS-15008/15009/15010/15012 (journeys + governed engagement + matching, over the existing `/opportunities` + `/workflows` + `/approvals` gates), and FS-15013 (editorial integrity, over the `/distribution` + `/reports` editorial gates). Each reconciles identity the ADR-0017 way when it builds a live registry.
- **Guardrail (unchanged from ADR-0017):** name-space reconciliation runs stricter than entity-space; anything touching human-authority or editorial contracts stays propose-only + human-confirmed; the canon never becomes a competing build authority.
